n8watch
Network Night Watch

Real-time IP and network host monitoring with a web dashboard, historical metrics, configurable alert rules, email notifications, and optional remote desktop access.

System Guide · Version 1.0 · Node.js 24 LTS

1 · System Overview

n8watch is a lightweight, self-hosted network monitoring system that continuously pings a configurable list of hosts, stores historical results in a local SQLite database, and pushes live metrics to a React web dashboard via WebSocket. No external database server is required — everything runs as a single Node.js process.

Key Features

  • Real-time ICMP monitoring — configurable ping cycles (count, interval, timeout) for every target.
  • Web dashboard — live status grid with latency, jitter, packet loss, and uptime badges.
  • Historical data — SQLite store with configurable retention (default 90 days); CSV export.
  • Alert engine — JavaScript expression conditions (e.g. packet_loss > 20) with per-rule cooldowns and per-target scoping.
  • Email notifications — HTML-formatted SMTP alerts with deduplication via cooldown timers.
  • Live config reload — edit config.yaml while running; targets and rules are applied within one second.
  • WebSocket push — all connected browsers receive updated metrics immediately after each ping cycle.
  • Multi-interface support — bind outgoing pings to a specific NIC per target.
  • System-defined & user-defined targets — permanent config-based hosts plus ad-hoc dashboard targets.
  • HTTPS support — optional TLS configuration for serving the dashboard securely.

Architecture at a Glance

Manual Installation

Step 1 — Clone the repository

git clone https://github.com/revul93/n8watch.git
cd n8watch

Step 2 — Run the interactive setup script

# Linux / macOS
bash setup.sh

# Windows (PowerShell)
powershell -ExecutionPolicy Bypass -File setup.ps1

The setup script performs the following steps automatically:

  1. Checks for Node.js v24 LTS and installs it (via nvm or the platform package manager) if missing.
  2. Installs backend Node.js dependencies (npm install --omit=dev).
  3. Installs and builds the React frontend (cd client && npm install && npm run build).
  4. Detects real network interfaces and writes them into config.yaml; falls back to copying config.example.yaml.
  5. Optionally installs PM2, starts the application, and configures system-boot auto-startup.

Step 3 — Manual setup (alternative to the script)

# Backend dependencies
npm install --omit=dev

# Frontend build
cd client && npm install && npm run build && cd ..

# Configuration
cp config.example.yaml config.yaml
# → edit config.yaml with your targets and SMTP settings
After installation, open config.yaml and configure your target IP addresses and SMTP credentials before starting the application.

3 · Configuration (config.yaml)

All settings live in config.yaml at the project root. The file is watched at runtime — save your changes and they are applied within one second without restarting the process.

3.1 General Settings

general:
  ping_interval: 30          # Seconds between ping cycles (1–59)
  ping_count: 5              # ICMP packets sent per cycle
  ping_timeout: 5            # Timeout in seconds per ping attempt
  data_retention_days: 90    # Days to keep historical ping results
  max_user_target_lifetime_days: 7  # Auto-expiry for dashboard-added targets
KeyDefaultDescription
ping_interval30Seconds between each full ping sweep of all targets
ping_count5Number of ICMP packets sent to each host per cycle
ping_timeout5Per-packet timeout in seconds
data_retention_days90Ping results older than this are automatically purged
max_user_target_lifetime_days7Days before a user-added target is automatically removed

3.2 Network Interfaces (optional)

interfaces:
  - name: "eth0"            # OS interface name (used with ping -I)
    alias: "Primary LAN"   # Human-readable label shown in the dashboard
    ipv4: "192.168.1.100"  # IPv4 address bound to this interface
  - name: "eth1"
    alias: "Management NIC"
    ipv4: "10.0.0.1"

When present, the dashboard shows an interface drop-down in the Add Target panel and displays a teal interface badge on each target card. Run ip link (Linux) or ifconfig (macOS) to list your interface names. The setup script auto-detects and populates this section.

3.3 Targets

targets:
  - name: "Google DNS"
    ip: "8.8.8.8"
    group: "DNS Servers"        # Optional grouping / filter label
    interface: "eth0"           # Optional: binds outgoing ping to this NIC
  - name: "Core Router"
    ip: "192.168.1.1"
    group: "Local Network"
    # interface: "eth1"         # Omit to use the default system interface
FieldRequiredDescription
nameYesDisplay name in the dashboard and reports
ipYesIPv4 address or resolvable hostname
groupNoLabel used to group and filter targets in the dashboard
interfaceNoOS interface name; must match an entry in the interfaces section

3.4 System-Defined vs User-Defined Targets

System-Defined Targets

Targets listed in config.yaml are system-defined. They are loaded at startup, kept in sync on every live config reload, and persist as long as they appear in the file.

User-Defined Targets

Any host added via the dashboard's Add Target panel (or via POST /api/targets/user-targets) is a user-defined target. These are temporary and expire automatically after max_user_target_lifetime_days. They can also be removed individually from the dashboard at any time.

Feature System-Defined User-Defined
Defined in config.yaml✅ Yes❌ No
Persists across restarts✅ Yes⚠️ Temporary
Live-reloaded with config✅ Yes❌ No
Added via dashboard / API❌ No✅ Yes
Subject to alert rules✅ Yes✅ Yes
Manually removable from dashboard❌ No✅ Yes
Tip: Use system-defined targets for critical, permanent hosts and user-defined targets for quick ad-hoc checks without touching config.yaml.

3.6 Alerts & Email Notifications

alerts:
  email_notifications: true   # Set false to disable all email alerts

  smtp:
    host: "smtp.gmail.com"
    port: 587
    secure: false             # true = port 465 (TLS); false = port 587 (STARTTLS)
    user: "you@gmail.com"
    pass: "app-password"      # Use an app-specific password
    from: "n8watch <monitor@yourdomain.com>"
    to:
      - "admin@yourdomain.com"

  rules:
    - name: "Host Down"
      condition: "packet_loss == 100"
      severity: "critical"    # critical | warning
      cooldown: 300           # Min seconds between repeated alerts for the same target
      # targets:              # Omit or leave empty to apply rule to ALL targets

    - name: "High Packet Loss"
      condition: "packet_loss > 20"
      severity: "warning"
      cooldown: 600
      targets:                # Restrict to specific targets by name, IP, or numeric ID
        - "Google DNS"

    - name: "High Latency"
      condition: "avg_latency > 200"
      severity: "warning"
      cooldown: 600

    - name: "High Jitter"
      condition: "jitter > 50"
      severity: "warning"
      cooldown: 600

Alert Rule Condition Variables

VariableTypeDescription
is_alive0 or 1Whether the host responded
packet_loss0–100 (%)Percentage of lost packets
avg_latencymsAverage round-trip time
min_latencymsMinimum round-trip time
max_latencymsMaximum round-trip time
jitterms(max − min) / 2
packets_sentintegerTotal packets sent in the cycle
packets_receivedintegerTotal packets received

Example conditions:

condition: "packet_loss == 100"   # Host completely unreachable
condition: "packet_loss > 20"     # More than 20% packet loss
condition: "avg_latency > 200"    # Latency exceeds 200 ms
condition: "jitter > 50"          # Jitter exceeds 50 ms
condition: "is_alive == 0"        # Host not responding

5 · Utility Scripts

n8watch ships two maintenance scripts in scripts/. Always stop the application before running either script to avoid database conflicts.

5.1 flush-data.js — Clear Rows

Deletes rows from the database without removing the database file. Useful for freeing disk space or resetting metrics without losing the schema. 

# Via npm shortcut (flushes ping results only — default)
npm run db:flush

# Via npm shortcut (flushes everything — ping results, alerts, and all targets)
npm run db:flush:all

# Direct invocation with flags
node scripts/flush-data.js --ping-results         # ping results only (default)
node scripts/flush-data.js --alerts               # alert records only
node scripts/flush-data.js --ping-results --alerts  # both
node scripts/flush-data.js --all                  # ping results + alerts + targets
node scripts/flush-data.js --all --yes            # non-interactive (skip confirmation)
FlagWhat is deleted
(none) or --ping-resultsAll rows in ping_results
--alertsAll alert records
--ping-results --alertsPing results and alert records
--allPing results, alerts, and all targets
--yesSuppresses the confirmation prompt

5.2 nuke-db.js — Complete Wipe & Recreate

Deletes the database file entirely (including WAL and SHM sidecar files) and recreates an empty schema from scratch. Auto-increment counters are reset; no data survives. 

# Via npm shortcut (interactive — requires typing YES)
npm run db:nuke

# Non-interactive (CI / automated scripts)
node scripts/nuke-db.js --yes
Warning: nuke-db.js is irreversible. All ping history, alert records, and target entries are permanently deleted. After nuking, start the application normally — targets from config.yaml will be reloaded on the next ping cycle.

7 · Optional: Remote Desktop Setup (Linux)

This guide shows how to run n8watch on a headless Linux server, display the dashboard full-screen in a browser, and access the screen remotely via VNC. This is ideal for a dedicated monitoring workstation or a wall-mounted display.

7.1 Install a Desktop Environment

sudo apt update
sudo apt install xfce4 xfce4-goodies lightdm -y

7.2 Configure Auto-Login

Edit the LightDM configuration to enable automatic login to the XFCE desktop:

sudo nano /etc/lightdm/lightdm.conf

Add the following block (replace administrator with your username):

[Seat:*]
xserver-command=X -core
autologin-user=administrator
autologin-session=xfce

7.3 Install Chromium

sudo apt install chromium-browser -y

7.4 Auto-Launch the Dashboard in Kiosk Mode

Create an autostart entry so Chromium opens the n8watch dashboard on every login:

mkdir -p ~/.config/autostart
nano ~/.config/autostart/browser.desktop

Paste the following (replace YOUR_PORT with your configured port, e.g. 3000):

[Desktop Entry]
Type=Application
Exec=chromium-browser --kiosk http://localhost:YOUR_PORT
Hidden=false
NoDisplay=false
X-GNOME-Autostart-enabled=true
Name=Monitoring

7.5 Install and Configure VNC for Remote Access

# Install x11vnc
sudo apt install x11vnc -y

# Set a VNC password
x11vnc -storepasswd

Create a systemd service so VNC starts automatically with the display manager:

sudo nano /etc/systemd/system/x11vnc.service

Paste the following (replace administrator with your username):

[Unit]
Description=Start x11vnc
After=display-manager.service

[Service]
Type=simple
ExecStart=/usr/bin/x11vnc -auth guess -forever -loop -noxdamage -repeat \
  -rfbauth /home/administrator/.vnc/passwd -rfbport 5900 -shared
User=administrator

[Install]
WantedBy=multi-user.target

Enable and start the service:

sudo systemctl daemon-reexec
sudo systemctl daemon-reload
sudo systemctl enable x11vnc
sudo systemctl start x11vnc

Connect from any VNC client (e.g. RealVNC, TigerVNC, Remmina) to <server-ip>:5900 using the password you set above.

7.6 Prevent Screen Sleep

To keep the display on at all times, disable power management in the XFCE Power Manager (xfce4-power-manager-settings → Display tab → turn all sleep/blank timers OFF), or apply the settings from the command line: 

xset s off       # Disable screen saver
xset -dpms       # Disable Display Power Management
xset s noblank   # Disable screen blanking
To make the xset commands persistent, add them as an autostart entry in ~/.config/autostart/ using the same [Desktop Entry] format as the browser entry above.

8 · Optional: Firewall Configuration

The following iptables rules provide a hardened baseline for a monitoring server — allowing only essential traffic and dropping everything else.

Persistence note: iptables rules are not persistent across reboots by default. To persist them, install iptables-persistent:
sudo apt install iptables-persistent then sudo netfilter-persistent save 
# ── 1. Flush all existing rules ────────────────────────────────────────────
iptables -F
iptables -X
iptables -t nat -F
iptables -t mangle -F

# ── 2. Set default policies ────────────────────────────────────────────────
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

# ── 3. Allow loopback traffic ──────────────────────────────────────────────
iptables -A INPUT -i lo -j ACCEPT

# ── 4. Allow established and related connections ───────────────────────────
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# ── 5. Allow SSH (prevent lockout) ─────────────────────────────────────────
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# ── 6. Allow n8watch dashboard (adjust port if changed in config.yaml) ─────
iptables -A INPUT -p tcp --dport 3000 -j ACCEPT

# ── 7. Allow VNC remote access (only if using the remote desktop setup) ────
iptables -A INPUT -p tcp --dport 5900 -j ACCEPT

# ── 8. Allow ICMP so the server itself is reachable ────────────────────────
iptables -A INPUT -p icmp -j ACCEPT
Tip: To restrict dashboard and VNC access to a trusted network only (e.g. your LAN 192.168.1.0/24), replace -j ACCEPT on those rules with -s 192.168.1.0/24 -j ACCEPT.

9 · Optional: Backup & Restore

n8watch ships an optional backup.sh shell script that archives all critical data — the SQLite database, the YAML configuration, and application logs — into a timestamped .tar.gz file. Backups can be restored to any compatible n8watch installation.

Important: Stop the application before restoring a backup to avoid database conflicts. Use npm run pm2:stop or Ctrl+C if running directly.

9.1 Usage

# Create a backup (saved to ./backups/ by default)
./backup.sh --backup

# Dry-run — shows what would be backed up without writing anything
./backup.sh --backup --dry-run

# Dry-run with a custom output directory
./backup.sh --backup --dry-run --output /path/to/dir

# Create a backup in a specific directory
./backup.sh --backup --output /path/to/dir

# Restore from an existing backup archive
./backup.sh --restore /path/to/file.tar.gz

# Show help
./backup.sh --help

9.2 Flags

FlagDescription
--backupCreate a compressed archive of the database, config, and logs
--restore <file>Restore database, config, and logs from a .tar.gz backup archive
--dry-runPrint what would be included in the backup without creating the archive
--output <dir>Override the default output directory (default: ./backups/)
--helpDisplay usage information

9.3 What Is Backed Up

ItemPathNotes
SQLite databasedata/n8watch.dbAll targets, ping results, and alert records
WAL / SHM sidecarsdata/n8watch.db-walIncluded when present
Configuration fileconfig.yamlYour current running configuration
Log fileslogs/out.log and error.log

9.4 Restore Procedure

# 1. Stop n8watch
npm run pm2:stop

# 2. Restore from archive
./backup.sh --restore /path/to/n8watch-backup-2025-01-15T12-00-00.tar.gz

# 3. Restart n8watch
npm run pm2:start
Warning: Restoring a backup overwrites the current database and config.yaml. Create a fresh backup of your existing state before restoring if you need the ability to roll back.
